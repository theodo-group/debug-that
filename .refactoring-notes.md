# Refactoring Notes — Session Polymorphism

## Behavioral Change: `isDapRuntime("bun")`

The old `isDapRuntime()` in entry.ts only excluded `undefined` and `"node"`:
```ts
return runtime !== undefined && runtime !== "node";
```

This meant `--runtime bun` would have been treated as a DAP runtime (incorrect — Bun uses CDP).
The new version explicitly excludes `"bun"` from DAP runtimes.

**Impact**: Only affects the theoretical case where someone passes `--runtime bun` explicitly.
In practice this flag is only used for `lldb`, `python`, `debugpy`, etc.
Bun is detected via the command binary name in `createAdapter()`, not via `--runtime`.

**Decision needed**: Is this behavioral fix acceptable? (It corrects a latent bug.)

## Behavioral Change: DapSession `getConsoleMessages`/`getExceptions` `--since` semantics

**Before (DapSession)**: `--since N` filtered messages by timestamp (`m.timestamp >= N`)
**After (BaseSession)**: `--since N` returns the last N messages (`slice(-N)`)

The BaseSession implementation matches the DebugSession (CDP) behavior, which treats `since` as a count.
The DapSession's timestamp-based filtering was inconsistent with CDP behavior.

**Impact**: DAP sessions (lldb, python) now use the same `--since` semantics as CDP sessions.
**Decision needed**: Is the unified "last N" behavior acceptable, or should `--since` remain timestamp-based for DAP?
